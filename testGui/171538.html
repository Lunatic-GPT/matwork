Hey all:

I've got a GUI already programmed that allows a user to
graphically filter time-based data structure. For example,
data structure X may have the following fields:

X.time
X.message_id
X.message_length
X.destination.net_number
X.destination.net_position
X.data_out

The user graphically selects the field he wants to filter
by, chooses the filter method, enters the numerical filter
constraints, clicks the OK button, and gets his desired answer.

The need has come up to be able to do this automatically
with predefined arguments (i.e., without the user needing
the GUI). I don't really want to write a separate program
to reduce the GUI's functionality to a command-line level
program. So I threw in some extra arguments to the GUI's
M-file in hopes to use the pre-existing code to do the job.
 My hopes were to use these extra arguments to pre-populate
the GUI's fields and run the GUI's logic to spit out the
answer without having the GUI appear.

I've figured out how to pass the arguments to the GUI's
M-file and pre-populate the uicontrols to said arguments.
However, I have not been able to figure out how to make the
GUI execute it's logic (normally done by pressing the OK
button). As things stand now, the user has to click the OK
button and everything runs fine after that. But this code
is to be used in my group's automation efforts.

Is there a way to make a GUI run one of its subfunctions
before it appears, calculate the desired answer, and return
control to the function that called it...all without
appearing on the screen?

BTW, I've already tried executing the OK button's callback
in the GUI's OpeningFcn.

Thanks for any help

- Will Slater

Either in or immediately after the figure() statement that
creates the figure, set the figure Visible property to be 'off'.
It will then not display on the screen. It will, though, still need
the screen to -exist- (e.g., Matlab not run with -nodesktop)
in order to create and position all of the controls, whose
handles the routine probably needs in order to function properly.

When you get to the point in the program logic where you would
like the simulated click to occur, manually call the Callback
of the OK button.

If the callback was specified as a cell array,

cb = get(okbutton, 'Callback');
cb{1}(okbutton, [], cb{2:end});

I figured it out. There's a block of code created by GUIDE
that initializes the GUI before the OpeningFcn is executed.
 It's demarked above and below by the warning "DO NOT EDIT".
 Ironically, this was the code I needed to edit to get the
desired result.

Originally, the initialization code created by GUIDE looks
like this:


% Begin initialization code - DO NOT EDIT
gui_Singleton = 1;
gui_State = struct('gui_Name', mfilename, ...
                   'gui_Singleton', gui_Singleton, ...
                   'gui_OpeningFcn', @MY_GUI_OpeningFcn, ...
                   'gui_OutputFcn', @MY_GUI_OutputFcn, ...
                   'gui_LayoutFcn', [], ...
                   'gui_Callback', []);
if nargin & isstr(varargin{1})
    gui_State.gui_Callback = str2func(varargin{1});
end

if nargout
    [varargout{1:nargout}] = gui_mainfcn(gui_State,
varargin{:});
else
    gui_mainfcn(gui_State, varargin{:});
end
% End initialization code - DO NOT EDIT


To get the result I desired, I added lines to the code as such:


% Begin initialization code - DO NOT EDIT
gui_Singleton = 1;
gui_State = struct('gui_Name', mfilename, ...
                   'gui_Singleton', gui_Singleton, ...
                   'gui_OpeningFcn', @MY_GUI_OpeningFcn, ...
                   'gui_OutputFcn', @MY_GUI_OutputFcn, ...
                   'gui_LayoutFcn', [], ...
                   'gui_Callback', []);
if nargin & isstr(varargin{1})
    gui_State.gui_Callback = str2func(varargin{1});
end

if nargout
    [varargout{1:nargout}] = gui_mainfcn(gui_State,
varargin{:});
    if nargin > 1
        tool_handle = findobj('Tag', 'MY_GUIs_Name');
        MY_GUI('OK_Button_Callback', tool_handle, ...
            [], guidata(tool_handle));
        uiresume
else
    gui_mainfcn(gui_State, varargin{:});
end
% End initialization code - DO NOT EDIT


That allowed me to feed extra inputs to the GUI, have it do
its thing, and return control to the program that called it.
 Hopefully this bit of my personal discovery can help anyone
else who might have this issue crop up.

Thanks to all who commented.

 - Will Slater
